# Data Flow 실행순서 계획서 초안

## 1. 문서 목적

본 계획서는 이미지 집합에 대해 다음 목표를 순차적으로 수행하는 운영 흐름을 정의한다.

1. 이미지별 캡션 및 구조화 메타데이터 생성
2. 이미지-텍스트 정합성 평가
3. 평가 결과에 따른 승인, 재작성, 감사 분기
4. 승인 결과의 메타데이터 파일 기록
5. 필요 시 캡션 기반 파일명 변경
6. 전체 실행 로그 및 산출물 저장

## 2. 시스템 역할 정의

### MainAgent

전체 배치를 제어하는 오케스트레이터다.
이미지 목록을 수집하고 정렬하며, 이미지마다 SubAgent를 1개씩 순차 실행한다.
평가 결과를 보고 승인 여부를 결정하고, 저장, 메타데이터 기록, 리네임을 지시한다.

### CaptionSubAgent

이미지 1장을 입력받아 `cv-mcp.image_metadata`를 1회 호출한다.
출력은 `alt_text`, `caption`, `metadata`다. `cv-mcp`는 이 목적에 맞게 단일 이미지 중심의 툴을 제공한다.

### EvalSubAgent

생성된 캡션을 독립적으로 검증한다.
`moondream-mcp`의 caption, VQA, object detection, visual pointing 계열 도구를 사용해 이미지 근거를 다시 수집한다. `moondream-mcp`는 이미지 캡셔닝, 질의응답, 객체 탐지, 위치 지정 기능을 제공한다.

### PersistenceAgent

filesystem MCP로 중간 산출물 JSON, 로그, 감사 대상 목록을 저장한다.
filesystem MCP는 디렉터리 조회, 파일 쓰기, 파일 이동을 지원한다.

### MetadataWriteAgent

승인된 결과를 ExifTool MCP로 파일 메타데이터에 기록한다.
ExifTool MCP는 150개 이상 포맷의 메타데이터 읽기, 쓰기, 삭제와 EXIF, IPTC, XMP 등 다중 표준을 지원한다.

## 3. 전체 실행순서 개요

```text
Batch Start
  -> Input Discovery
  -> Preflight Validation
  -> For each image in sorted order:
       -> CaptionSubAgent
       -> EvalSubAgent
       -> Decision Gate
           -> accept  -> persist -> write metadata -> rename(optional)
           -> rewrite -> retry generation/eval
           -> audit   -> persist audit package
           -> error   -> persist error and continue
  -> Batch Summary
  -> Final Report
```

## 4. 단계별 Data Flow 초안

## Phase 0. 입력 준비

### Step 0-1. 작업 디렉터리 확인

MainAgent가 filesystem MCP로 허용 디렉터리와 입력 폴더를 확인한다.
출력은 `input_dir`, `output_dir`, `audit_dir`, `log_dir` 경로다.

### Step 0-2. 이미지 목록 수집

입력 폴더에서 대상 이미지 파일 목록을 읽는다.
지원 확장자 정책은 별도 설정 파일로 유지한다.

### Step 0-3. 실행 순서 정렬

기본 정렬 기준은 파일명 오름차순이다.
옵션으로 생성일, 수정일, EXIF 촬영일 기반 정렬 정책을 둘 수 있다.

### Step 0-4. 배치 manifest 생성

각 이미지에 대해 아래 초기 레코드를 만든다.

```json
{
  "sequence_no": 1,
  "image_path": "/input/img_0001.jpg",
  "status": "queued"
}
```

이 manifest는 배치 재실행 시 체크포인트 역할을 한다.

## Phase 1. 이미지별 생성

### Step 1-1. MainAgent가 이미지 1장 선택

MainAgent는 정렬된 목록에서 아직 완료되지 않은 첫 이미지를 선택한다.

### Step 1-2. CaptionSubAgent 생성

해당 이미지 전용 SubAgent를 생성한다.
SubAgent는 오직 이미지 1장만 컨텍스트로 받는다.

### Step 1-3. `cv-mcp.image_metadata` 호출

SubAgent는 다음 호출을 수행한다.

```json
{
  "tool": "cv-mcp.image_metadata",
  "args": {
    "file_path": "/input/img_0001.jpg",
    "mode": "double"
  }
}
```

`cv-mcp`는 `image_metadata(image_url|file_path, caption_override?, config_path?) -> { alt_text, caption, metadata }`를 제공하며, README상 `double`은 vision 단계로 alt와 caption을 만들고 text 단계로 metadata를 생성한다.

### Step 1-4. 생성 결과 정규화

SubAgent는 결과를 아래 공통 포맷으로 반환한다.

```json
{
  "sequence_no": 1,
  "image_path": "/input/img_0001.jpg",
  "status": "generated",
  "alt_text": "...",
  "caption": "...",
  "metadata": { ... }
}
```

### Step 1-5. 1차 저장

filesystem MCP로 `output/intermediate/0001.generated.json`에 저장한다.
filesystem MCP의 `write_file`은 새 파일 생성과 기존 파일 overwrite를 지원한다.

## Phase 2. 이미지별 평가

### Step 2-1. EvalSubAgent 생성

MainAgent는 같은 이미지에 대해 별도 EvalSubAgent를 만든다.
이 SubAgent는 생성 결과와 이미지 경로를 입력받는다.

### Step 2-2. 독립 evidence 추출

EvalSubAgent는 `moondream-mcp`를 사용해 최소 3종의 근거를 모은다.

1. 독립 상세 caption
2. VQA 응답
3. 객체 탐지 또는 위치 지정 결과

예시 질의:

- 이 이미지에 보이는 핵심 객체는 무엇인가?
- 사람 수는 몇 명인가?
- 읽을 수 있는 텍스트가 있는가?
- 생성된 caption에서 언급된 객체 X가 실제 있는가?

`moondream-mcp`는 `caption_image`, `query_image`, `detect_objects`, `point_objects` 도구를 제공한다.

### Step 2-3. 평가용 atomic unit 생성

생성된 caption과 metadata에서 atomic unit를 추출한다.

예:

- object: dog
- attribute: black
- relation: dog_on_sofa
- count: 2 people
- text: SALE
- scene: beach
- action: running

### Step 2-4. grounding 평가

EvalSubAgent는 atomic unit별로 이미지 근거 존재 여부를 판정한다.

산출:

- precision
- recall
- f1
- hallucination rate

### Step 2-5. holistic judge 평가

별도 judge rubric으로 다음을 채점한다.

- accuracy
- completeness
- conciseness
- relevance

### Step 2-6. 최종 판정 산출

EvalSubAgent는 아래 형태로 반환한다.

```json
{
  "sequence_no": 1,
  "image_path": "/input/img_0001.jpg",
  "status": "evaluated",
  "scores": {
    "precision": 0.86,
    "recall": 0.78,
    "f1": 0.82,
    "hallucination": 0.14,
    "judge_accuracy": 0.88,
    "judge_completeness": 0.80,
    "judge_conciseness": 0.84,
    "judge_relevance": 0.90,
    "final": 0.84
  },
  "decision": "accept"
}
```

### Step 2-7. 평가 결과 저장

filesystem MCP로 `output/intermediate/0001.evaluated.json`에 저장한다.

## Phase 3. 의사결정 게이트

### Step 3-1. accept

조건 예시:

- precision >= 0.75
- judge_accuracy >= 0.80
- final >= 0.80

처리:

- 영구 저장
- 메타데이터 기록 단계로 이동
- 필요 시 리네임

### Step 3-2. rewrite

조건 예시:

- 세부 정확도는 부족하지만 복구 가능
- hallucination이 경미
- 누락이 많음

처리:

- `rewrite_count += 1`
- 최대 1~2회 재생성
- 재생성 시 prompt 보정 또는 `caption_override` 전략 도입 가능
- `cv-mcp`는 `caption_override`를 받아 비전 캡션 단계를 건너뛰고 metadata를 만들 수 있다.

### Step 3-3. audit

조건 예시:

- judge는 높은데 grounding은 낮음
- count, text, object에서 충돌
- OCR 또는 개수 판단이 불안정

처리:

- 감사 패키지 저장
- 수동 검토 큐로 이동
- 다음 이미지로 진행

### Step 3-4. error

조건 예시:

- MCP 호출 실패
- 파일 손상
- 응답 파싱 실패

처리:

- error 레코드 저장
- 다음 이미지 진행
- 배치 전체는 중단하지 않음

## Phase 4. 승인 결과 영구 저장

### Step 4-1. canonical result 작성

승인된 이미지는 최종 레코드로 묶는다.

```json
{
  "sequence_no": 1,
  "image_path": "/input/img_0001.jpg",
  "alt_text": "...",
  "caption": "...",
  "metadata": { ... },
  "scores": { ... },
  "decision": "accept",
  "version": "v1"
}
```

### Step 4-2. 결과 파일 저장

filesystem MCP로 `output/final/0001.result.json`에 저장한다.
필요하면 배치 전체 집계 파일도 업데이트한다.

## Phase 5. 메타데이터 기록

### Step 5-1. 기록 대상 태그 매핑

예시 매핑:

- `caption` -> XMP:Description
- `alt_text` -> IPTC:Caption-Abstract 또는 XMP 계열 대체 태그
- `metadata.tags` -> XMP:Subject
- `metadata.scene` -> XMP:Category 또는 custom namespace

구체 태그 스키마는 운영 정책으로 별도 정의한다.

### Step 5-2. ExifTool MCP 호출

MetadataWriteAgent가 ExifTool MCP를 통해 실제 파일 메타데이터를 기록한다.
ExifTool MCP는 읽기, 쓰기, 삭제와 다중 메타데이터 표준을 지원한다.

### Step 5-3. 기록 검증

기록 직후 메타데이터를 다시 읽어 반영 여부를 검증한다.
불일치 시 `metadata_write_failed` 상태를 남기고 원본 파일은 유지한다.

## Phase 6. 파일명 변경

### Step 6-1. 파일명 후보 생성

정책 예시:

```text
{sequence_no}_{caption_slug}.jpg
```

예:

```text
0001_woman_walking_snowy_city_street.jpg
```

### Step 6-2. 금지 규칙 적용

- 최대 길이 제한
- OS 예약문자 제거
- 중복 시 suffix 부여
- 개인식별 가능 정보 포함 금지

### Step 6-3. filesystem MCP로 rename

filesystem MCP의 `move_file`로 파일명을 바꾼다.
README 기준 `move_file`은 source를 삭제하는 이동형 동작이다.

### Step 6-4. 경로 재기록

최종 result JSON의 `image_path`를 새 경로로 업데이트한다.

## Phase 7. 배치 종료 처리

### Step 7-1. 통계 집계

집계 항목:

- total
- accept
- rewrite
- audit
- error
- metadata_write_failed
- rename_failed

### Step 7-2. 배치 요약 보고서 생성

filesystem MCP로 `output/reports/batch_summary.json`과 `batch_summary.md` 저장

### Step 7-3. 감사 대상 목록 생성

`audit` 이미지 목록만 분리 저장

## 5. 실행순서 기준 상세 상태 전이

```text
queued
  -> generating
  -> generated
  -> evaluating
  -> evaluated
  -> accept_pending | rewrite_pending | audit_pending | error
  -> persisted
  -> metadata_written
  -> renamed
  -> completed
```

실제로는 `metadata_written`과 `renamed`는 선택 단계이므로, 아래 두 종료 경로를 허용한다.

```text
accept -> persisted -> completed
accept -> persisted -> metadata_written -> renamed -> completed
```

## 6. 실패 처리 정책

### 생성 실패

- 동일 이미지 1회 재시도
- 반복 실패 시 `error`

### 평가 실패

- 독립 evaluator만 1회 재시도
- 반복 실패 시 `audit`

### 메타데이터 기록 실패

- 결과 JSON은 유지
- 원본 파일은 유지
- `metadata_write_failed`

### 리네임 실패

- 메타데이터와 결과 JSON은 유지
- 기존 파일명 유지
- `rename_failed`

## 8. 권장 MCP 세트와 역할 요약

- `cv-mcp`: 이미지 1장 기준 alt_text, caption, metadata 생성
- `moondream-mcp`: 독립 caption, VQA, object detection 기반 검증
- filesystem MCP: 목록 수집, JSON 저장, 파일 이동, 리네임
- ExifTool MCP: EXIF, IPTC, XMP write-back

## 9. 바로 구현 가능한 최소 운영 버전

초기 MVP는 아래 순서만 먼저 구현하면 된다.

```text
1. input discovery
2. image_metadata generation
3. independent evaluation
4. decision save
5. metadata write-back
```

파일명 변경은 2차 단계로 미뤄도 된다.
운영 리스크는 보통 리네임보다 평가와 메타데이터 매핑에서 먼저 나온다.
