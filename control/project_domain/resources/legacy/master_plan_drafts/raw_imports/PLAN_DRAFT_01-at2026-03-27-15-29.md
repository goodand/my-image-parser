# 이미지 캡션 파이프라인 Data Flow 계획서 초안

## 1. 목적

본 파이프라인의 목적은 이미지 파일을 개별 단위로 처리하면서, 각 이미지에 대해 다음 작업을 안정적으로 수행하는 것이다.

- 이미지별 독립 SubAgent 실행
- 이미지 캡션 생성
- 생성된 캡션의 영속 저장
- 메타데이터 반영 여부 추적
- 파일명 변경 후보 관리
- 전체 작업 상태의 중앙 관리

구조 원칙은 다음과 같다.

- Skills는 SubAgent orchestration 전용
- MCP는 데이터 저장과 상태 확인의 기준점
- SubAgent의 자유형 응답은 참고용이며, 완료 판정은 MCP 조회 결과로만 수행
- 이미지 1개 = 작업 단위 1개 = SubAgent 1개

Codex는 Skills를 반복 가능한 워크플로 패키지로, MCP를 외부 도구 및 공유 시스템 연결로, Subagents를 병렬 위임 수단으로 설명한다. 또한 custom agent는 별도 역할을 가지도록 설계할 수 있다.

## 2. 구성요소

### 2.1 Orchestrator Skill

부모 Skill이 전체 실행을 담당한다.

역할:

- 입력 이미지 목록 수집
- 이미지별 job/subtask 생성
- SubAgent 실행
- 재시도 정책 적용
- 최종 집계
- 커밋 단계 승인

### 2.2 Execution Management MCP

SubAgent 실행 상태를 관리한다.

권장 후보:

- `agent-task-manager-mcp`

이 MCP는 task/subtask 생성, 상태 업데이트, 락, 세션, 체크포인트를 지원하므로 “어떤 이미지가 어느 단계까지 처리됐는가”를 관리하기 좋다.

### 2.3 Image Registry MCP

이미지 경로, 캡션, 상태, rename 후보를 저장하는 영속 레지스트리다.

권장 후보:

- `Context Portal (ConPort)`

ConPort는 구조화된 프로젝트 컨텍스트 저장용 MCP이며, custom data와 progress 같은 구조화 정보를 저장하는 용도에 잘 맞는다.

### 2.4 Worker MCPs

실제 작업 수행용 MCP들이다.

예:

- 이미지 caption 생성 MCP
- EXIF/metadata write MCP
- filesystem MCP

MCP 연결과 설정은 Codex의 `config.toml` 또는 CLI의 `codex mcp add`로 관리할 수 있다.

{

  "source_path": "/images/a.jpg",
  "current_path": "/images/a.jpg",
  "caption": null,
  "caption_status": "pending",
  "metadata_status": "pending",

  "rename_candidate": null,

## 4. 실행 순서별 Data Flow

## Phase 0. 환경 준비

### Step 0-1. MCP 연결

Codex 프로젝트 설정에 다음 MCP를 등록한다.

- execution management MCP
- image registry MCP
- caption MCP
- metadata MCP
- filesystem MCP

### Step 0-2. Skill 및 SubAgent 등록

- 부모 orchestration Skill 등록
- `image-caption-worker` custom subagent 등록

## Phase 1. 입력 등록

### Step 1-1. 입력 이미지 목록 수집

부모 Skill이 처리 대상 이미지 목록을 수집한다.

입력 예:

```json
[
  {"source_path": "/images/a.jpg"},
  {"source_path": "/images/b.jpg"}
]
```

### Step 1-2. image_id 발급

각 이미지에 고유 `image_id`를 부여한다.

예:

- `img_000001`
- `img_000002`

### Step 1-3. Registry 초기 레코드 생성

각 이미지마다 Image Registry MCP에 초기 레코드를 저장한다.

초기 상태:

- `caption_status = pending`
- `metadata_status = pending`
- `rename_status = pending`

### Step 1-4. Execution Record 생성

Execution Management MCP에 각 이미지별 subtask를 만든다.

초기 상태:

- `phase = caption`
- `state = queued`
- `attempt = 0`

이 단계 이후부터 부모 Skill은 “작업 대상 목록”을 로컬 메모리로 들고 있기보다, MCP를 기준으로 진행 상태를 조회한다.

## Phase 2. SubAgent 실행 준비

### Step 2-1. 실행 가능한 작업 조회

부모 Skill이 Execution MCP에서 `queued` 상태인 작업을 조회한다.

### Step 2-2. 작업 락 획득

중복 실행 방지를 위해 job/subtask에 락을 건다.

### Step 2-3. 상태 변경

해당 작업 상태를 다음으로 전이한다.

- `state: queued -> running`
- `started_at` 기록
- `attempt += 1`

## Phase 3. SubAgent 캡션 생성

### Step 3-1. Worker 호출

부모 Skill이 `image-caption-worker` SubAgent를 실행한다.

SubAgent 입력은 최소화한다.

예:

```json
{
  "job_id": "job_20260326_001",
  "image_id": "img_000123"
}
```

핵심은 경로와 기존 상태를 입력 본문에 길게 넣지 않는 것이다.
Worker는 필요한 데이터는 MCP에서 조회한다.

### Step 3-2. Worker가 Registry 조회

Worker는 Image Registry MCP에서 `image_id`로 다음 정보를 읽는다.

- `source_path`
- `current_path`
- 기존 `caption`
- 기존 상태

### Step 3-3. Worker가 caption MCP 실행

Worker는 `current_path`를 기반으로 caption MCP를 호출해 캡션을 생성한다.

출력 예:

```json
{
  "caption": "a brown dog running on grass"
}
```

### Step 3-4. rename 후보 생성

Worker는 캡션을 slugify하여 rename 후보를 만든다.

예:

- caption: `a brown dog running on grass`
- rename candidate: `a-brown-dog-running-on-grass.jpg`

이 단계에서는 실제 rename을 하지 않는다.
후보만 만든다.

### Step 3-5. Worker 결과 저장

Worker는 결과를 Registry MCP에 먼저 기록한다.

갱신 필드:

- `caption`
- `caption_status = completed`
- `rename_candidate`
- `updated_at`

### Step 3-6. Execution 상태 갱신

Worker는 Execution MCP에 다음을 기록한다.

- `phase = caption`
- `state = completed`
- `finished_at`
- `evidence = [caption, registry_update_reference]`

이 구조의 핵심은 SubAgent 텍스트 응답보다 MCP 기록이 먼저 진실 원천이 되는 것이다.

## Phase 4. 부모 검증

### Step 4-1. 부모가 Worker 응답 수신

부모 Skill은 SubAgent의 자연어 응답 또는 JSON 응답을 받는다.

하지만 이 값만으로 완료 처리하지 않는다.

### Step 4-2. Registry 재조회

부모 Skill은 Image Registry MCP를 조회하여 실제로 다음이 기록됐는지 확인한다.

- `caption != null`
- `caption_status = completed`
- `rename_candidate != null`

### Step 4-3. Execution 재조회

부모 Skill은 Execution MCP를 조회하여 다음을 확인한다.

- `state = completed`
- `evidence` 존재
- `finished_at` 존재

### Step 4-4. 완료 판정

둘 다 만족할 때만 해당 이미지 작업을 “caption phase 완료”로 인정한다.

불일치 시:

- `state = failed`
- `last_error` 기록
- 재시도 큐로 이동

## Phase 5. 후속 작업 분기

caption 완료 후에는 세 갈래로 분기 가능하다.

### Branch A. 캡션만 저장

가장 안전한 기본 모드다.

결과:

- Registry에는 캡션 저장 완료
- 파일은 원본 유지
- 메타데이터 미반영

### Branch B. 메타데이터 반영

Worker 또는 후속 worker가 metadata MCP를 호출하여 caption을 EXIF/IPTC/XMP에 쓴다.

완료 후 Registry 갱신:

- `metadata_status = completed`

실패 시:

- `metadata_status = failed`
- `last_error` 기록

### Branch C. 파일명 변경

filesystem MCP로 rename을 수행한다.

권장 방식:

1. collision check
2. 최종 rename 수행
3. `current_path` 갱신
4. `rename_status = completed`

## Phase 6. 커밋 단계

### Step 6-1. 커밋 대상 조회

부모 Skill이 다음 조건을 만족하는 이미지들을 조회한다.

- `caption_status = completed`
- `metadata_status in (pending, completed)`
- `rename_status in (pending, completed)`

### Step 6-2. 승인 정책 적용

커밋 전략은 2가지다.

#### 자동 커밋 모드

캡션 생성 즉시 메타데이터/rename까지 수행

#### 수동 승인 모드

부모 Skill이 검토 목록을 만들고 승인된 것만 후속 반영

초기 운영은 수동 승인 모드가 더 안전하다.

### Step 6-3. 최종 반영 후 Registry 동기화

모든 파일 반영 후 최종 상태를 Registry에 기록한다.

최종 예:

```json
{
  "image_id": "img_000123",
  "current_path": "/images/a-brown-dog-running-on-grass.jpg",
  "caption": "a brown dog running on grass",
  "caption_status": "completed",
  "metadata_status": "completed",
  "rename_status": "completed"
}
```

## Phase 7. 집계 및 보고

### Step 7-1. 전체 작업 요약

부모 Skill이 MCP 조회 결과만으로 집계한다.

예:

- 총 이미지 수
- caption 완료 수
- metadata 완료 수
- rename 완료 수
- 실패 수

### Step 7-2. 실패 목록 추출

실패 이미지에 대해 다음을 출력한다.

- `image_id`
- `current_path`
- `phase`
- `last_error`
- `attempt`

### Step 7-3. 재시도 목록 작성

재시도 정책 예:

- caption 실패: 최대 2회
- metadata 실패: 최대 1회
- rename 실패: 충돌 해소 후 수동 재시도

## 5. 권장 운영 규칙

### 규칙 1. SubAgent 입력은 최소화

SubAgent에는 `job_id`, `image_id`만 넘긴다.

### 규칙 2. MCP가 항상 기준

완료 여부는 SubAgent 응답이 아니라 MCP 상태 조회로 판정한다.

### 규칙 3. 파일 변경은 마지막

rename과 metadata write는 캡션 저장이 끝난 뒤에만 수행한다.

### 규칙 4. path와 caption은 한 저장소에서 관리

이미지 경로와 캡션의 authoritative store는 한곳으로 통일한다.
이 용도에는 ConPort 같은 구조화 저장소가 잘 맞는다.

### 규칙 5. 실행 이력은 별도 보관

작업 실행 이력, 시도 횟수, 체크포인트는 execution MCP가 담당한다.
이 용도에는 `agent-task-manager-mcp`가 적합하다.

## 6. 추천 아키텍처 결론

가장 안정적인 초안은 이 구조다.

### Skill

- 부모 orchestration skill
- image-caption-worker subagent skill

### MCP

- Execution 관리: `agent-task-manager-mcp`
- Image registry: `ConPort`
- 실제 작업: caption MCP, metadata MCP, filesystem MCP

즉 흐름은 다음 한 줄로 요약된다.

**Skill이 SubAgent를 실행하고, 모든 결과 판정과 데이터 관리는 MCP에서 한다.**
