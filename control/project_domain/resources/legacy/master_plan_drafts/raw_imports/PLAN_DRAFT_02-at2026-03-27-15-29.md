# Data Flow 실행순서 계획서 초안

## 1. 목적

Canva Presentation을 이미지 저장소처럼 활용하여, 원본 프레젠테이션에서 이미지를 추출하고, 각 이미지에 대한 캡션 및 관련 문서 매핑 정보를 정제한 뒤, 이를 기반으로 새로운 Presentation 파일을 생성하는 파이프라인을 정의한다.

본 문서는 특히 다음 두 가지를 명확히 하는 데 목적이 있다.

1. 시스템 자동 처리 구간과 사람 검토 구간의 경계
2. 각 실행 단계별 입력, 처리, 출력, 재실행 기준

## 2. 범위

본 계획서는 아래 흐름을 포함한다.

Canva presentation 선택
→ presentation file 다운로드
→ presentation file에서 이미지 추출
→ 이미지 초안 캡션 생성
→ 사람 검토를 통한 캡션 확정
→ 이미지 임베딩
→ 관련 문서 후보 검색
→ reranker 기반 Top-5 후보 생성
→ 사람 검토를 통한 문서 매핑 확정 또는 이상치 분류
→ 이상치에 대한 수기 설명 작성
→ 이미지 + 최종 캡션 + 최종 설명 기반 새 presentation 생성

다음 항목은 이미 구현 완료된 것으로 가정하되, 전체 Data Flow 상에는 포함한다.

- presentation file에서 이미지 추출
- Gemini 기반 이미지 캡션 생성
- 이미지 + 이미지 캡션 기반 presentation 생성

## 3. 기본 전제

### 3.1 소스 전제

원본 데이터는 Canva Presentation이다.
Canva는 읽기 및 export 용도로만 사용하며, 역방향 동기화는 고려하지 않는다.

### 3.2 검수 전제

사람 검토는 다음 다섯 가지 업무를 수행한다.

- 이미지 캡션 적합 여부 검토
- 부적합한 캡션 직접 수정
- reranker Top-5 문서 매핑 후보 검토
- 문서 매핑 확정 또는 이상치(outlier) 라벨 지정
- 이상치 이미지에 대한 설명서 직접 작성

### 3.3 재처리 전제

최종 캡션이 수정되면, 해당 캡션을 retrieval 입력으로 사용하는 경우 이후 검색 및 rerank 결과가 달라질 수 있으므로 관련 단계는 재실행 가능해야 한다.

## 4. 주요 데이터 객체

파이프라인에서 관리해야 하는 핵심 객체는 아래와 같다.

### 4.1 Job

한 번의 실행 단위를 의미한다.
원본 Canva design 1건을 기준으로 1개의 Job을 생성한다.

예시 필드:

- `job_id`
- `source_design_id`
- `source_design_title`
- `source_export_file`
- `status`
- `created_at`
- `pipeline_version`

### 4.2 Image Item

원본 프레젠테이션에서 추출된 개별 이미지 단위다.

예시 필드:

- `image_id`
- `job_id`
- `source_page_no`
- `source_element_order`
- `image_path`
- `image_hash`
- `is_duplicate`
- `is_decorative`

### 4.3 Caption Record

이미지별 캡션 생성 및 검토 결과다.

예시 필드:

- `image_id`
- `draft_caption`
- `approved_caption`
- `caption_review_status`
- `caption_edited_by_human`
- `caption_last_updated_at`

### 4.4 Mapping Record

이미지와 관련 문서 간의 매핑 상태를 저장한다.

예시 필드:

- `image_id`
- `retrieval_query`
- `candidate_doc_ids`
- `reranked_top5`
- `selected_doc_id`
- `mapping_status`
- `outlier_label`
- `manual_description`

## 5. 실행순서 기준 Data Flow

## Step 0. Job 생성 및 실행 준비

### 입력

- 사용자가 지정한 Canva presentation
- 또는 검색 조건으로 선택된 Canva presentation
- 검색 대상 문서 저장소
- 파이프라인 설정값

### 처리

- `job_id` 생성
- 작업 디렉토리 및 메타데이터 파일 생성
- 실행 이력 저장용 manifest 초기화

### 출력

- `job_manifest.json`
- 작업 폴더 구조
- Job 상태: `initialized`

## Step 1. Canva Presentation 식별 및 다운로드

### 입력

- Canva design ID 또는 검색 조건

### 처리

- Canva MCP를 이용해 대상 presentation 검색 또는 지정
- 대상 design 선택
- presentation 파일 export
- 원본 파일 및 메타데이터 저장

### 출력

- `source_presentation.pptx`
- `source_metadata.json`
- Job 상태: `source_downloaded`

### 비고

이 단계는 Canva와의 유일한 연결 지점이며, 이후 단계는 모두 로컬 또는 내부 파이프라인에서 처리한다.

## Step 2. Presentation 파일에서 이미지 추출

### 입력

- `source_presentation.pptx`

### 처리

- 슬라이드별 이미지 추출
- 이미지별 고유 ID 부여
- 페이지 번호, 순서, 해시값 기록
- 필요 시 중복 이미지 또는 장식성 요소 필터링

### 출력

- `/images/` 폴더
- `image_manifest.jsonl`
- Job 상태: `images_extracted`

### 비고

장식성 아이콘, 배경 조각, 너무 작은 이미지 등은 이후 검토 부담을 줄이기 위해 제외 옵션을 둘 수 있다.

## Step 3. 이미지 초안 캡션 생성

### 입력

- 추출된 이미지 목록

### 처리

- 각 이미지에 대해 캡션 초안 생성
- 초안 캡션과 생성 시점 메타데이터 기록

### 출력

- `caption_draft.jsonl`
- 상태: 각 이미지별 `caption_drafted`

### 비고

이 단계는 기존 완료 프로세스를 그대로 재사용한다.

## Step 4. 사람 검토 1차: 이미지 캡션 검토 및 수정

### 입력

- 이미지
- 초안 캡션

### 처리

검토자는 각 이미지에 대해 다음 중 하나를 수행한다.

1. 초안 캡션 승인
2. 초안 캡션 수정 후 승인

필요 시 검토 코멘트를 남긴다.

### 출력

- `caption_review.jsonl`
- 각 이미지별 최종 캡션 확정
- 상태:
  - `caption_approved`
  - 또는 `caption_edited`

### 중요 규칙

이 단계가 끝난 뒤의 `approved_caption`만 이후 검색과 최종 산출에 사용한다.
즉, 초안 캡션은 참고용이고 운영 기준값은 최종 승인 캡션이다.

## Step 5. Retrieval 입력 생성

### 입력

- 이미지 원본
- 승인된 캡션

### 처리

- 검색용 입력 생성
- 필요 시 이미지 임베딩 생성
- 필요 시 승인 캡션을 포함한 텍스트 검색 질의 생성
- 사용 모델 버전과 생성 시점 기록

### 출력

- `image_embedding.parquet` 또는 유사 포맷
- `retrieval_input.jsonl`
- 상태: `retrieval_ready`

### 비고

이 단계에서 어떤 조합을 사용할지는 구현 정책으로 정한다.

가능한 방식은 다음과 같다.

- 이미지 임베딩만 사용
- 승인 캡션 텍스트만 사용
- 이미지 임베딩 + 승인 캡션 결합 사용

실무적으로는 세 번째 방식이 가장 관리하기 좋다.

## Step 6. 관련 문서 후보 검색

### 입력

- `retrieval_input.jsonl`
- 문서 저장소 인덱스

### 처리

- 이미지별 관련 문서 후보 검색
- 충분한 후보군 확보를 위해 Top-K 후보 추출
- 후보 점수 및 검색 근거 저장

### 출력

- `retrieval_candidates.jsonl`
- 상태: `candidates_retrieved`

### 비고

여기서의 Top-K는 rerank 이전의 넉넉한 후보군이다.
예를 들어 20~50개 수준으로 둘 수 있다.

## Step 7. Reranker 기반 Top-5 후보 생성

### 입력

- 검색 후보군
- 이미지
- 승인 캡션

### 처리

- 각 이미지에 대해 후보 문서 재정렬
- 최종 검토용 Top-5 생성

### 출력

- `reranked_top5.jsonl`
- 상태: `reranked`

### 비고

사람이 보는 최종 후보 리스트는 이 단계 결과를 기준으로 한다.

## Step 8. 사람 검토 2차: Top-5 문서 매핑 검토

### 입력

- 이미지
- 승인 캡션
- reranker Top-5 문서 후보

### 처리

검토자는 각 이미지에 대해 다음 중 하나를 수행한다.

1. Top-5 중 1개 문서를 최종 매핑으로 선택
2. 적절한 문서가 없다고 판단하여 이상치(outlier)로 라벨링

### 출력

- `mapping_review.jsonl`
- 상태:
  - `mapping_selected`
  - 또는 `outlier_labeled`

### 비고

이 단계에서는 “후보 순위의 적절성”과 “최종 선택의 타당성”을 분리해서 기록하는 것이 좋다.
즉, Top-5 품질 자체에 대한 평가 필드가 있으면 향후 reranker 개선에 도움이 된다.

## Step 9. 사람 검토 3차: 이상치 이미지 설명 작성

### 입력

- `outlier_labeled` 상태의 이미지
- 승인 캡션
- 검토자 메모

### 처리

- 검토자가 해당 이미지에 대한 설명 또는 문서성 텍스트를 직접 작성
- 이후 새 presentation에 들어갈 최종 설명으로 저장

### 출력

- `manual_description.jsonl`
- 상태: `manual_description_done`

### 비고

이 단계는 “문서 매핑이 없지만 presentation에는 포함되어야 하는 이미지”를 처리하기 위한 예외 경로다.

## Step 10. 최종 콘텐츠 조립

### 입력

- 이미지
- 승인 캡션
- 선택된 문서 또는 수기 설명

### 처리

각 이미지별 최종 표현 단위를 생성한다.

- 매핑이 확정된 경우
  - 선택된 관련 문서 기반 설명 생성 또는 첨부
- 이상치인 경우
  - 수기 설명 사용

이 단계에서 슬라이드 단위 콘텐츠 블록을 확정한다.

### 출력

- `final_content_manifest.jsonl`
- 상태: `content_assembled`

## Step 11. 새 Presentation 파일 생성

### 입력

- 최종 콘텐츠 manifest
- 이미지 파일
- 승인 캡션
- 최종 설명

### 처리

- 새 presentation 구조 생성
- 이미지, 캡션, 설명 배치
- 파일 렌더링 및 저장

### 출력

- `output_presentation.pptx`
- 또는 원하는 최종 presentation 포맷
- Job 상태: `rendered`

### 비고

이 단계는 기존 완료 프로세스를 확장하여 재사용할 수 있다.

## Step 12. 최종 검증 및 산출물 확정

### 입력

- 최종 presentation 파일
- 전체 manifest

### 처리

- 누락 이미지 존재 여부 확인
- caption 승인 누락 여부 확인
- outlier인데 설명 미작성 상태인지 확인
- 최종 산출물 패키징

### 출력

- `output_presentation.pptx`
- `final_job_report.json`
- Job 상태: `completed`

## 6. 사람 개입 지점 요약

사람이 반드시 수행하는 단계는 아래 3개 구간이다.

### 검토 구간 A. 캡션 검토

- 초안 캡션 확인
- 부적합 시 직접 수정
- 승인 캡션 확정

### 검토 구간 B. 문서 매핑 검토

- reranker Top-5 후보 검토
- 최종 문서 선택 또는 이상치 지정

### 검토 구간 C. 이상치 설명 작성

- 적절한 매핑 문서가 없는 이미지에 대해 수기 설명 작성

## 7. 상태 흐름 초안

이미지 단위 상태는 아래 순서로 이동한다.

`extracted`
→ `caption_drafted`
→ `caption_approved` 또는 `caption_edited`
→ `retrieval_ready`
→ `candidates_retrieved`
→ `reranked`
→ `mapping_selected` 또는 `outlier_labeled`
→ `manual_description_done`
→ `finalized`

주의할 점은 `mapping_selected`와 `manual_description_done`은 동시에 발생하지 않는다는 점이다.

- 문서 매핑이 있으면 `mapping_selected`로 종료
- 이상치이면 `outlier_labeled` 후 `manual_description_done`으로 이동

## 8. 재실행 규칙

운영상 가장 중요한 부분이다.

### 8.1 원본 presentation이 바뀐 경우

Step 1부터 전체 재실행한다.

### 8.2 이미지 추출 로직이 바뀐 경우

Step 2부터 전체 재실행한다.

### 8.3 캡션이 수정된 경우

Step 5부터 재실행한다.
즉, retrieval input 생성, 후보 검색, rerank 결과를 다시 계산한다.

### 8.4 문서 매핑만 변경된 경우

Step 10부터 재실행한다.
최종 콘텐츠 조립과 presentation 생성만 다시 수행하면 된다.

### 8.5 이상치 설명만 수정된 경우

Step 10부터 재실행한다.

## 9. 권장 산출물 구조

아래처럼 Job 단위 폴더를 구성하면 운영이 편하다.

```text
/job/{job_id}/
  source/
    source_presentation.pptx
    source_metadata.json
  images/
    *.png
  manifests/
    image_manifest.jsonl
    caption_draft.jsonl
    caption_review.jsonl
    retrieval_input.jsonl
    retrieval_candidates.jsonl
    reranked_top5.jsonl
    mapping_review.jsonl
    manual_description.jsonl
    final_content_manifest.jsonl
    final_job_report.json
  output/
    output_presentation.pptx
```

## 10. 최종 운영 원칙

이 파이프라인의 운영 기준값은 아래 세 가지다.

첫째, 최종 승인 캡션만 검색과 산출에 사용한다.
둘째, 문서 매핑이 실패한 이미지는 outlier로 분리하고 수기 설명으로 처리한다.
셋째, 최종 presentation 생성은 사람이 확정한 값만 사용한다.

즉, 자동화는 초안을 만들고 후보를 제시하는 역할까지 담당하고,
최종 의미 결정은 사람 검토를 거쳐 확정하는 구조로 설계한다.
